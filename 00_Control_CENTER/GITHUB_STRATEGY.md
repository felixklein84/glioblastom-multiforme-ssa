# GITHUB_STRATEGY.md

## Purpose

This file defines how my GitHub profile, repositories, pinned projects, repository descriptions, README files, and portfolio connections should be structured.

The goal is to make my GitHub look like a curated professional technical portfolio, not a random collection of student projects, experiments, and unfinished repositories.

GitHub should support:

- job applications
- master applications
- technical interviews
- networking
- personal branding
- project credibility
- portfolio website traffic

---

## Core Principle

GitHub is not storage.

GitHub is public proof of work.

Every public repository should have a reason to exist.

Every pinned repository should strengthen my technical positioning.

The profile should communicate:

```text
I can model complex systems, build simulations and data workflows, understand real-world constraints, and present technical work clearly.
```

---

## Target Impression

When someone opens my GitHub, they should think:

- This person has real technical depth.
- This person is not just uploading coursework.
- This person can structure messy problems.
- This person understands mathematical modeling and implementation.
- This person has taste in documentation and visual presentation.
- This person can connect technical work to real-world applications.
- This person could be useful in robotics, simulation, data science, automation, or technical product roles.

---

## What GitHub Should Not Look Like

Avoid the impression of:

- random student repositories
- unfinished coursework
- generic data science tutorials
- messy old code dumps
- unclear project names
- empty READMEs
- no screenshots
- no installation instructions
- no project story
- private or sensitive files
- too many pinned repositories
- inflated claims without proof

## Repository Categories

Classify every repository into one of these categories.

### Flagship

Represents my best work.

Can be pinned, shown in portfolio, and discussed in interviews.

### Showcase

Good public project, supports profile, but not necessarily top pinned.

### Supporting

Useful public repository but not central to my story.

### Private

Should remain private because of sensitive data, rough code, or limited public value.

### Archive

Old project kept for reference, not actively maintained, not part of public profile.

---

## Public / Private Decision Rules

Make a repository public only if:

- it does not contain sensitive data
- the README explains the project clearly
- the project name is understandable
- the code or output demonstrates something useful
- limitations are documented
- it does not weaken my profile

Keep private if:

- it contains company data
- it contains raw university or thesis material that should not be public
- it contains private notes
- it is too messy and not worth cleaning
- it reveals personal information
- it cannot be explained clearly

Archive if:

- it is old
- it is not useful
- it is not worth improving
- it distracts from stronger work

---

## Tone Rules

Use:

- precise
- calm
- confident
- technical
- direct
- honest

Avoid:

- hype
- fake depth
- vague motivation
- startup exaggeration
- overuse of AI language
- student-project apologies

Good phrasing:

```text
This project implements...
The simulation models...
The workflow supports...
The current version focuses on...
Limitations include...
Future work could include...
```

Bad phrasing:

```text
This innovative project leverages cutting-edge technology...
I am passionate about...
This revolutionary system transforms...
This project changes the future of...
```
